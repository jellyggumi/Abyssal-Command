## Architectural boundaries and ownership

| Boundary | Contractual responsibility | Must not own |
|---|---|---|
| `RulesKernel` | Validate a rule-versioned command, reduce one encounter state, expose terminal outcome, and reject invalid input without mutation. | DOM, engine APIs, time, randomness, storage, network, audio, assets, pricing, or telemetry. |
| `CampaignConfig` | Supply only the pinned schedule IDs, ordered intents, and per-encounter rule data from the immutable rule contract. | Adapter-specific defaults, procedural generation, or altered values. |
| `ReplayVerifier` | Canonically order valid replay records, execute the same reducer boundary twice, and compare canonical outputs. | Silent repair, device-specific substitutions, or presentation. |
| `SettlementBoundary` | Accept exactly five terminal outcomes and return the pinned settlement result. | Persistence, entitlement, catalog, pricing, or later-stage transfer. |
| `InputAdapter` | Normalize browser pointer/touch, keyboard, assistive activation, and mobile input into one versioned command-record shape. | Rule arithmetic, outcome rewriting, or different device rules. |
| `PresentationAdapter` | Render state, errors, intent labels, and non-color-only alternatives from kernel output. | Calculating state or inferring unreturned outcomes. |
| `AudioAssetAdapter` | Request optional, named assets only after local user intent; expose unavailable/failed audio as non-blocking. | Rule state, outcome, command legality, or an unapproved provider route. |
| `PersistenceAdapter` | Store/load an explicit versioned snapshot only when later approved; validate before restoring. | Treating unvalidated data as state, or silently fixing malformed data. |
| `TransportAdapter` | Remain absent by default; if later approved, carry immutable envelopes and report failures explicitly. | Multiplayer, database, provider, or protocol selection in this P3 artifact. |
| `HostAdapter` | Bind the presentation adapter to static browser, mobile shell, or a future engine/renderer. | Altering `CampaignConfig`, reducer ordering, or replay semantics. |

The `RulesKernel`, `CampaignConfig`, `ReplayVerifier`, and `SettlementBoundary` form the deterministic core. All other boundaries are dependency-inverted adapters. Any adapter replacement must preserve the core’s versioned input/output contract.

## Deterministic compatibility contract

### Command, reducer, and replay envelopes

A command record has the pinned shape:

```json
{"rules_version":"stage1-rules-v1","command":"STRIKE|BRACE|DISRUPT|RECOVER","tick":0,"sequence":1}
```

`RulesKernel.reduce(previous_state, record)` MUST return either an accepted next state or `{accepted:false, reason:<stable-code>, state:previous_state}`. For rejection, `state` MUST be byte-equivalent under canonical serialization to the supplied state: no mutation, substitution, retry, clamping, reordered trace, or fabricated command is allowed.

`ReplayVerifier.replay(encounter_id, records)` MUST resolve the selected pinned schedule, order records by `(tick, sequence)`, detect duplicate sequences, pass each record through the same reducer, and retain each result. `ReplayVerifier.validate(encounter_id, records)` MUST perform two independent replays and compare canonical state serialization. A mismatch is an explicit result, not a state rewrite.

`SettlementBoundary.settle(outcomes)` MUST accept exactly five terminal outcomes in campaign order. It MUST reject any incomplete, extra, non-terminal, or malformed list without applying a partial settlement. The values and conversion rule remain solely those pinned in `rule-contract-v1.md`.

### Required reducer invariants

- `rules_version` is exact; commands, foe intents, outcomes, ticks, sequences, schedule, round coherence, bounds, and transient flags are validated before state transition.
- `sequence` is positive and unique per replay. `tick` equals the current zero-based round for a live reduction.
- The reducer resolves the legal player command, checks immediate victory before adverse resolution, then applies one scheduled adverse effect, clears transient guard/counter state, and applies the pinned terminal priority.
- The visible labels **Press the threat**, **Hold the line**, **Break the pattern**, and **Regroup and reassess** map only to `STRIKE`, `BRACE`, `DISRUPT`, and `RECOVER`; they do not create alternate commands or outcomes.
- The only allowable campaign for this version is the five-ID schedule above. A later campaign or rules version is a separately versioned configuration, not fallback behavior.

### Invalid-input and error semantics

Stable machine-readable error codes MUST distinguish at least: malformed state, state rules-version mismatch, malformed schedule, invalid bounds, schedule/round incoherence, foe-intent mismatch, stale transient state, malformed record, record rules-version mismatch, tick mismatch, invalid sequence, duplicate sequence, unknown command, unmet command precondition, terminal-state command, non-array replay, and invalid settlement list.

Errors MUST be safe to display through a presentation adapter as a plain-language status without exposing provider credentials, stored data, stack traces, or untrusted payload content. An adapter MAY offer a new explicit player action after rejection; it MUST NOT synthesize a valid event or repair state.

## Static-web, mobile, accessibility, and media boundaries

The observed source surface contains a static `index.html` module entry, `app.js`, and `game-core.js`; the observed GitHub Pages workflow publishes an allowlisted static closure. A static-host adapter MUST require no server, identity, database, multiplayer, or runtime provider to dispatch a command. Technical constraints require public GitHub Pages web/PWA evidence before Android packaging is considered; this document does not authorize either activity.

A mobile or browser `InputAdapter` MUST route pointer, touch, keyboard, switch/assistive activation, and any host-native tap through the same command-record factory and reducer. It MUST expose command availability and rejection status through text, preserve keyboard focus behavior, and never rely on color, motion, icon, sound, or a raw enum as the sole signal. Audio is optional; audio failure or mute state MUST not suppress a textual command, status, intent, counter, or outcome.

`AudioAssetAdapter` and `PresentationAdapter` MUST receive immutable kernel output and may add only non-authoritative feedback. Asset lookup failures, unsupported media, denied autoplay, offline availability, missing locale text, and renderer failure MUST result in an observable degraded state that leaves the deterministic state unchanged.

## Storage, network, and security constraints

- Persistence is optional and disabled by default at this contract boundary. A later approved adapter must serialize an explicit `rules_version`, campaign position, records, state, outcomes, and settlement; it must validate the complete snapshot through the kernel before use.
- A malformed, stale, tampered, unsupported, or schema-incompatible snapshot MUST be rejected with an explicit error and an explicit choice to start a new session. It MUST NOT be partially loaded or silently normalized.
- Network and transport are optional and absent by default. No gameplay action may depend on a provider response, provider timing, or client secret.
- Adapters MUST not embed secrets, make ambient external calls, execute untrusted data as code, inject untrusted content into markup, or log command/state payloads to an unapproved endpoint.
- Host-visible data minimization applies: command/replay data stays local unless a later approved adapter specifies purpose, retention, consent, failure mode, and owner. No payment, entitlement, catalog, or offer path may affect a command, schedule, timing, access, information, or outcome.

## Measurable integration acceptance criteria

A later authorized integrator can satisfy this contract only by producing evidence that all of the following are true:

1. **Configuration integrity:** loading the rules version yields exactly `S1-E1` through `S1-E5` in order; each intent list equals this document’s table; any schedule deviation returns a stable rejection and changes no state.
2. **Five-encounter replay:** for each of the five IDs, a legal record corpus replays twice to identical canonical state; out-of-order input has the same ordered replay result; duplicate sequence remains a recorded rejection; no malformed event changes state.
3. **Reducer parity:** all four commands pass one shared reducer boundary from browser pointer/touch, keyboard, assistive activation, and mobile-tap adapters. For the same versioned records, every adapter yields byte-identical canonical state and trace.
4. **Settlement integrity:** exactly five terminal outcomes settle according to the pinned rules; four or six outcomes, a non-terminal outcome, or a malformed list returns an explicit error and no partial settlement.
5. **Accessibility and media degradation:** command label, availability, intent, rejection, and outcome remain text-observable with audio disabled, assets unavailable, color removed, motion reduced, and an assistive activation route; no optional media failure alters core state.
6. **Adapter isolation:** static-host, mobile-host, audio/assets, persistence, transport/network, and renderer/engine adapters do not import rule arithmetic or mutate kernel state; unapproved external-provider invocation is absent.
7. **Snapshot safety:** an approved future persistence adapter rejects a malformed or mismatched snapshot before presentation and does not silently repair it. This criterion does not approve persistence itself.
8. **Scope integrity:** the evidence names no engine, renderer, framework, database, multiplayer model, provider, pricing path, or deployment decision.

New commands, schedules, campaign length, values, providers, persistence semantics, multiplayer, database, renderer, or engine choices require separately authorized, hash-pinned artifacts and cannot be introduced through adapter fallback.
