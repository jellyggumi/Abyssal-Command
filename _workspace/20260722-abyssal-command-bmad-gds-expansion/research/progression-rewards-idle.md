# Progression, Reward Certainty, and Idle Catch-up Research Packet

## Research question

For an offline, no-commerce, movement-first action-survivor, how should guaranteed rewards, player choices, disclosed chance, collection progress, return timing, and offline catch-up create legible growth without manufacturing frustration or eroding tactical parity over **10–20 sessions**?

## Product boundary used in this packet

Abyssal Command is a single-player, mobile-first defense survivor: XP creates run-only skill choices; elite extraction unlocks permanent companions; boss wins unlock stages; a failed run resets run skills; and campaign completion is the Stage 10 boss. Its state is deterministic at 60 Hz, local/offline, and has no accounts, network, cloud, multiplayer, or commerce. Permanent companion acquisition and run-build state are explicitly distinct. [S1]

**Non-negotiable design boundary:** this packet does not recommend monetization. There are no purchases, premium currencies, ads, paid skips, social-pressure mechanics, or pay-to-accelerate alternatives. Any system that makes a player feel they must return at a particular time to avoid loss is treated as a fairness failure, not a retention objective.

## Comparison: reward surfaces and fair implementation choices

| Surface | What the evidence supports | Fair no-commerce application | Reject / do not import |
| --- | --- | --- | --- |
| **Guaranteed cadence** | Competence needs a legible relationship between action and progress (S2–S3). The product already ties stage opening to boss success and persistent unlocks to extraction (S1). | Put the primary permanent track on visible, finite milestones: elite encounter → extraction progress → named companion/Archive unlock. A loss may cost run build, but it must advance a visible attempt/proficiency record when the player achieved the relevant encounter condition. | A permanent unlock whose only meaningful route is indefinite luck; resetting progress to stimulate repeated attempts; making a loss yield no information and no forward motion. |
| **Choice rewards** | Autonomy can improve engagement (S2–S3), but overload is conditional on complexity, difficulty, preference uncertainty, and goals (S4). | Offer compact, tactically distinct run choices with a one-line consequence and an inspectable current-build interaction. Keep choices run-bound so the choice changes play rather than permanently compounding power. | A large “choice” catalog at a stressful moment, opaque synergy math, false choices with a dominant option, or choices that convert into a permanent power lead. |
| **Probability disclosure** | S5 identifies a research line titled “Uncertainty Increases Resource Investment in the Process of Reward Pursuit”; commercial evidence shows disclosures can be obscure and weakly surfaced (S6). | Best default: make tactic-affecting permanent collection **certain**, eliminating the need for odds. If any noncombat surprise remains, show the complete outcome set and exact chance before commitment, make duplicates impossible until the set is exhausted, and never charge or time-gate another attempt. | Hidden tables, misleading rarity colors, variable-ratio “try again” loops, chance changing with undisclosed state, pseudo-near-misses, or probability disclosure buried in settings. “Odds displayed” is not a fairness substitute (S6). |
| **Collection progression** | Run-to-meta progression is a recognizable action-roguelite pattern in the official example (S9), while SDT warns against controlling/external-only motivation (S2). | Use W-05 **Archive** as a visible collection ledger: known silhouettes, exact unlock condition, current progress, and tactical role. Unlocking expands viable loadout choices; it must not create a mandatory strictly better companion. | Completion pressure based on limited windows, rarity scarcity, duplicate conversion currencies, collection score leaderboards, or a collection that turns the first 10–20 sessions into a raw-stat grind. |
| **Return timers / idle progress** | Idle games can make participation optional/redundant (S8), which conflicts with this product’s movement-first tactical core (S1). | Treat absence as a convenience layer, not the main progression engine: bounded, deterministic local catch-up may grant only a planning/Archive preparation token or a small fixed research increment. It may never simulate combat, roll extraction, unlock a stage, resolve a boss, or select a run skill. | “Your vault is full—return now,” loss for being absent, notification pressure, short caps optimized for check-ins, paid/earned speed-ups, or idle gains that exceed active tactical learning. |
| **Catch-up boundary and determinism** | The current contract requires local persistence and deterministic 60 Hz state updates (S1); S8 warns that delegation can displace meaningful agency. | Record a declared elapsed interval in the save, calculate a fixed capped grant once at resume, store the result, and show the formula/cap. Device clock anomalies should yield the safe minimum (zero grant), not punishment or a broken save. | Background combat simulation with unstored randomness, retroactive timer changes, repeated collection by reopening, “offline failure,” or catch-up that alters a deterministic combat replay. |

## Repeated patterns

1. **Legible contingency beats opaque intensity.** The supported motivation framing centers autonomy and competence (S2–S3); the product already has clear run/permanent boundaries (S1). Primary rewards should therefore answer “what did I do, what changed, and what can I do next?” on the result screen.
2. **Choices need bounded information, not maximal option count.** S4 finds overload is moderated rather than universal. The useful translation is a small set of inspectable, tactically non-identical choices—not a claim that fewer options are always better.
3. **Uncertainty requires restraint.** S5’s verified title frames uncertainty as increasing resource investment in reward pursuit, while S6 documents poor practical transparency in commercial random reward systems. In a fair no-commerce product, uncertainty is appropriate for momentary encounter texture, not for withholding permanent tactical capability.
4. **Idle mechanics trade agency for delegated progress.** S8 identifies the genre’s central tension. Because Abyssal’s intended mastery is movement positioning and build choice (S1), offline progress must stay auxiliary and cannot replace an active run.
5. **Visible collection can make permanent growth comprehensible without raw power inflation.** The collection ledger should state tactical identity and unlock conditions, preserving meaningful loadout breadth rather than escalating stats.

## Contradictions, risks, and anti-patterns

- **Uncertainty versus fairness:** S5’s title supports uncertainty as a resource-investment mechanism, while S6 shows probability disclosures are often insufficiently prominent in real commercial systems. Resolution: do not tune a permanent progression loop for repeated uncertain pursuit. Make permanent tactical rewards guaranteed; reserve disclosed surprise for nonessential presentation, if at all.
- **Choice versus decision fatigue:** S2–S3 support autonomy, but S4 says choice overload rises under specific complexity/difficulty/uncertainty conditions. Resolution: a player can defer to a recommended option without penalty, but the recommendation must explain why and remain a choice.
- **Idle comfort versus movement-first mastery:** S8’s optional participation is structurally at odds with S1’s player-controlled movement. Resolution: idle catch-up cannot produce combat wins, extraction resolution, stage clearing, or run-level skill selection.
- **Extraction as currently described versus fair pacing:** The contract says an elite kill creates an extraction “chance” (S1) but gives no disclosed success rule. That is a **product-spec gap**, not evidence of unfairness. Before implementation, define whether it is deterministic skill gating, transparent chance with a finite guarantee, or a nonrandom progress meter. The first and third are safer under this packet.
- **Offline local time is mutable:** Local/offline persistence has no trusted server clock (S1). Fighting clock changes with punitive lockouts harms legitimate offline users and complicates determinism. Grant only a small capped convenience reward, store the one-time calculation, and safely decline anomalous elapsed time.
- **Coercive patterns incompatible with this product:** paid skips/currencies, scarcity sales, streak-loss, daily login chains, social comparison, near-miss theatrics, hidden odds, duplicate dilution, short-cap “come back now” loops, and boss/collection timers that disadvantage time away. These are excluded even if they might raise repeat visits.

## Original, measurable design implications

These are **hypotheses/targets**, not claims proven by the sources.

1. **Guaranteed permanent-progress bound — target.** From the first eligible elite encounter, a player receives a visible, nonresetting extraction/Archive increment on every eligible completion and reaches its associated permanent companion or alternative tactical unlock in **at most 3 eligible completions**. The UI states `current / 3`, reward identity, and the condition before the next run. This protects competence after a failed extraction attempt without a pity-timer’s hidden math.
2. **Run-choice clarity — target.** Each XP threshold presents **3** options maximum, each described by (a) one effect, (b) one tagged tactical role, and (c) a before/after value where numerical. A recommended option may appear only with a visible build-based reason. In a moderated comprehension test, **≥80%** of participants must correctly predict the chosen option’s immediate tactical effect before confirming it.
3. **Permanent parity envelope — target.** Across scripted seed replays and human playtests, median Stage 1–10 completion time and median boss clear time for a **10-session** versus **20-session** account should remain within **±15%** when both use an appropriate unlocked companion/loadout. Permanent progression may widen strategy availability, but may not produce an automatic combat-stat superiority that invalidates positioning.
4. **No-probability combat collection — target.** **100%** of permanent, tactic-affecting companion/Archive unlocks follow a disclosed deterministic condition or finite progress meter; **0%** depend on hidden probabilities, duplicates, near misses, or a purchase/return opportunity. If a noncombat surprise table is ever retained, show all outcomes, exact odds, seed-independent rules, and an exhaustion guarantee before selection.
5. **Offline catch-up cap and agency boundary — target.** Catch-up is calculated once at resume from stored elapsed time, capped at **12 hours** per absence, and grants **≤20%** of the median permanent-progress value earned in one completed active run. It grants no combat XP, stage clear, extraction success, boss resolution, companion unlock, or run skill. Resume screen shows elapsed time accepted, cap, formula, and grant; negative/implausible local-clock deltas grant **0** without destroying progress.

## Candidate evaluation plan

| Question | Method | Measures | Pass condition (all are targets) |
| --- | --- | --- | --- |
| Is progression perceived as fair after bad luck or a loss? | 12–16 moderated sessions using identical encounter seeds; compare visible finite-progress treatment with an undisclosed-chance prototype only as a research control, never a ship candidate. | “I knew what advanced,” “I could recover,” “I felt pressured to retry” (5-point agreement); eligible-completion count to unlock. | ≥80% select 4–5 for the first two statements; ≤10% select 4–5 for pressure; no participant exceeds the 3-completion permanent-progress bound. |
| Are XP choices meaningful rather than overwhelming? | Counterbalanced 3-option versus 5-option prototype; no other balance changes. | Correct effect prediction, decision time, deferral, regret/confidence, and post-run skill-use distribution. | 3-option version keeps ≥80% effect prediction and reduces deferral/regret without collapsing selection of a single option above 70% across comparable build contexts. |
| Does permanent growth preserve 10–20-session tactical parity? | Run fixed seeds with accounts synthesized at session 10 and 20; then repeat with human pilot logs. | Completion time, boss clear time, damage taken, movement/hazard errors, selected companion/loadout, and build diversity. | Both time metrics remain within ±15%; build diversity does not shrink; movement/hazard error remains a meaningful predictor of outcome in both cohorts. |
| Does catch-up feel like convenience rather than obligation? | Simulate 0 h, 2 h, 12 h, 24 h, clock-back, and repeated-reopen resumes on exported local saves. | One-time grant correctness, duplicate grant count, replay determinism, participant pressure rating, and active/idle permanent-value ratio. | Zero duplicate grants; deterministic save/reload outcome; 0 benefit from clock-back; ≤20% active/idle ratio threshold; ≤10% report obligation (4–5/5). |
| Is the reward model implementation-transparent? | Design QA checklist plus a player “tell us the rule” test before content expansion. | Whether every persistent reward has a displayed condition/progress; whether any reward contains chance; UI-to-simulation rule match. | 100% persistent rewards pass disclosure/condition inspection; 0 hidden chance tables; no UI/simulation mismatch across deterministic replay cases. |

## Recommended concept direction

**Use a two-layer fair growth model.** Active runs deliver frequent, bounded, three-way build choices and skill-demonstrated encounter progress; the Archive records visible finite steps toward permanent *breadth*, not escalating combat power. A loss ends the temporary build but leaves a legible permanent trace only when the player performed the documented eligible action. Optional offline catch-up is small, formula-visible, local-only preparation—not autonomous combat.

This direction preserves Abyssal canon: **W-01 Echo** can frame run feedback, **W-02 Bind** can name companion affinity/loadout relationships, **W-03 Gate Integrity** can expose finite extraction conditions, **W-04 Domain** can distinguish stage tactics, and **W-05 Archive** can show the collection ledger. These are original labels and implications; no characters, assets, named content, or mechanics are copied from studied products.
