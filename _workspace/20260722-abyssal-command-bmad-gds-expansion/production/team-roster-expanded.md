# Abyssal Command expanded studio roster

**Planning state:** concept-validation only. This is an ownership and hand-off contract, not an implementation assignment or a gate decision. Every G1–G8 result remains **NOT MEASURED / NOT PASSED**; numeric values below are **UNVERIFIED TARGETS** copied from `qa/gate-measurements.md`.

## Scope and stage contract

| Stage | Scope | Public-beat contribution | Exit evidence owner |
|---|---|---|---|
| **1 — concept / presentation / core loop** | Define the bounded Cinder Span Command Feedback slice, its legible observer surfaces, and factual W-01…W-05 language. | An internally playable, **target 90-second** Cinder Span loop: threat, health, W-03 Gate Integrity, critical/cooldown state, movement objective, choice, and factual result hand-off. | Game QA assembles gate evidence; the production director records disposition. |
| **2 — balance / rewards / growth / encounters** | Specify and measure equal-budget rules, active-only reward/idle fairness, map/wave variation, and encounter pressure. | The same beat gains comparable offer/reward choices and replayable authored-seed map/wave variation; no new campaign route or player-facing archetype is implied. | Game QA with systems/progression, level/PCG, playtest-research, and programmer evidence. |
| **3 — combat feedback / resources / performance / QA / release** | Integrate observer-only feedback, build assets, controls/accessibility, local telemetry, performance, release and rollback proof. | The beat is readable in normal, mute, sound-off, and reduced-motion modes, then is eligible for a release decision only after the named evidence exists. | Performance/release, accessibility QA, game QA, then production director. |

**Binding implementation boundary:** `defense-catalog.js` remains the sole authored rules authority; the deterministic core remains fixed at **60 Hz**. `MapPlan` and `WavePlan` are immutable after admission. HUD, renderers, VFX, audio, narration, haptics, accessibility, telemetry, and export are detached observers: failure or mode changes must leave canonical state byte-identical. The game remains offline/local: no account, commerce, cloud sync, remote telemetry, background combat, or gameplay-time provider call. Stage 10 is terminal.

## Coordination interfaces and non-overlap rules

| Interface | Producer → consumer | Immutable hand-off | Non-overlap rule |
|---|---|---|---|
| **Rules / observer envelope** | Game programmer → feedback, narrative, audio, VFX, UI/UX, telemetry | Versioned detached observer event with event ID, simulation tick, authoritative fields, and no command channel. | Consumers never write rule state, advance RNG, mutate plan, persist campaign, reorder input, or retry gameplay. |
| **Catalog-plan admission** | Game designer + level/PCG → game programmer + QA | Catalog-approved `MapPlan`/`WavePlan` tuple, seed/key, pressure range, selection trace, and objective schema before tick 0. | Level/PCG may choose approved variants only; it cannot add/reorder objectives, alter rewards/campaign consequences, replace a failed seed, or extend Stage 10. |
| **Balance and reward change control** | Systems/progression → game designer + PM → programmer + QA | Equal-budget ledger, hypothesis, paired-seed fixture specification, and signed change request. | Systems/progression neither edits the canonical rules authority nor declares a G2/G5 result; programmer implements only signed scope. |
| **Presentation payload allowlist** | Narrative + combat-feedback → audio/VFX/UI/UX → accessibility QA | Caption/static semantic, event mapping, priority, fallback, asset ID, and mode behavior. | Presentation owns interpretation and assets, never outcomes; narrative cannot invent lore/history, consume core RNG, or imply a Stage 11 route. |
| **Build asset boundary** | Audio + technical art → performance/release → QA | Pinned local asset manifest, provenance/rights review, hash, fallback behavior, and source scan. | ElevenLabs or any provider is optional build-time generation only; no runtime SDK, credential, URL, request, or gameplay dependency. |
| **Study and telemetry evidence** | Playtest research + accessibility QA → game QA → director | Preregistered protocol, local consent boundary, fixture/capture IDs, raw-to-derived trace, exclusions, and evidence export. | Researchers report results and uncertainty; only Game QA updates the ledger, and only the director records the cycle disposition. |
| **Release / rollback** | Performance/release + QA → PM + director | Pinned build tuple, G6 evidence, network-disabled/export-delete proof, rollback steps, and known defects. | A passing implementation test, an isolated benchmark, or a research target cannot authorize release or any G1–G8 PASS. |
